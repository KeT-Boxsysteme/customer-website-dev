════════════════════════════════════════════════════════════════════════════════
  SERVER.JS – HAUPTSERVER
════════════════════════════════════════════════════════════════════════════════

  Startet die Express-Anwendung und verdrahtet alle Teile miteinander.

  Was hier passiert (der Reihe nach):
  1. Lädt alle Umgebungsvariablen aus .env
  2. app.set('trust proxy', 1) – WICHTIG für Render/Cloud-Deployment:
     Render terminiert SSL am Load Balancer, Node sieht intern HTTP.
     Ohne diesen Eintrag werden Session-Cookies mit secure:true nie gesetzt
     → Login schlägt in Produktion still fehl. Muss VOR der Session-Middleware stehen.
  3. Setzt EJS als Template-Engine (alle Views sind .ejs-Dateien)
  4. Aktiviert Middleware: Formular-Parser, JSON-Parser, statische Dateien (public/)
  5. Session-Verwaltung: Nutzer bleibt 24 Stunden eingeloggt (Cookie)
     secure: true nur in NODE_ENV=production (erfordert HTTPS)
  6. Flash-Nachrichten: Kurze Erfolgs-/Fehlermeldungen zwischen Seitenaufrufen
  7. Globale Variablen für alle Views: user (eingeloggter User), success, error
  8. Routen werden eingebunden – alle außer /auth benötigen Login (requireAuth)

  Routen-Übersicht:
    GET  /              → Weiterleitung zu /auth/login oder /dashboard
    GET  /dashboard     → Startseite nach Login (erfordert Login)
    /auth/...           → Login, Registrierung, Passwort (routes/auth.js)
    /users/...          → Userverwaltung (routes/users.js)
    /boxes/...          → Boxenverwaltung (routes/boxes.js)
    /monitoring/...     → Monitoring (routes/monitoring.js)
    /diagrams/...       → Diagramme (routes/diagrams.js)

  Fehlerseiten:
    404 → views/errors/404.ejs (Seite nicht gefunden)
    500 → views/errors/500.ejs (Serverfehler)

  CONFIG/DATABASE.JS – DATENBANKVERBINDUNG
════════════════════════════════════════════════════════════════════════════════

  Stellt eine Verbindung zur MSSQL-Datenbank her (Azure SQL).
  Verwendet einen Connection-Pool (max. 10 gleichzeitige Verbindungen).
  Die Verbindung wird nur einmal aufgebaut und danach wiederverwendet.

  Timeout-Konfiguration:
    connectionTimeout: 8000 ms  → Fehler nach 8 Sek. wenn DB nicht erreichbar
    requestTimeout:   15000 ms  → Einzelne Queries laufen max. 15 Sek.
  (Reduziert auf 8 Sek. damit Login-Fehler schnell angezeigt werden,
   statt 30 Sek. zu hängen wenn Azure DB pausiert ist.)

  Funktionen:
  ┌─────────────────┬────────────────────────────────────────────────────────┐
  │ getPool()       │ Gibt den aktiven DB-Pool zurück, stellt Verbindung her  │
  │                 │ falls noch keine besteht. Wird von allen Models genutzt.│
  ├─────────────────┼────────────────────────────────────────────────────────┤
  │ closePool()     │ Schließt den Pool sauber (z.B. beim Server-Shutdown).   │
  └─────────────────┴────────────────────────────────────────────────────────┘

  Benötigte .env-Variablen:
    DB_SERVER    → Adresse des MSSQL-Servers (z.B. ket-dev-website.database.windows.net)
    DB_DATABASE  → Name der Datenbank
    DB_USERNAME  → Datenbankbenutzer
    DB_PASSWORD  → Datenbankpasswort

  HINWEIS Azure SQL: Die Datenbank pausiert sich im kostenlosen/Serverless-Tier
  automatisch nach Inaktivität. Im Azure Portal unter
  "SQL Databases → Compute + Storage" auf "Provisioned" wechseln um das zu verhindern.

  MIDDLEWARE
════════════════════════════════════════════════════════════════════════════════

  ── middleware/auth.js ─────────────────────────────────────────────────────

    requireAuth(req, res, next)
      Prüft ob ein User in der Session eingeloggt ist.
      → Wenn nicht: Weiterleitung zu /auth/login mit Fehlermeldung.
      → Wenn ja: Anfrage wird durchgelassen (next()).
      Wird in server.js auf alle geschützten Routen angewendet.

  ── middleware/authorize.js ────────────────────────────────────────────────

    Definiert 4 Rollen und ihre Berechtigungen:
    ┌────────────────┬────────────────────────────────────────────────────┐
    │ admin          │ Zugriff auf alles (inkl. Userverwaltung)            │
    │ controller     │ Zugriff auf Boxen, Monitoring, Diagramme            │
    │ user           │ Nur Monitoring und Diagramme                        │
    │ box_user       │ Nur Monitoring und Diagramme (für Android-Tablet)   │
    └────────────────┴────────────────────────────────────────────────────┘

    authorize(...roles)
      Gibt eine Middleware-Funktion zurück, die prüft ob der eingeloggte
      User eine der erlaubten Rollen hat.
      → Wenn nicht: HTTP 403 und views/errors/403.ejs wird angezeigt.
      Beispiel: authorize(ROLES.ADMIN) erlaubt nur Admins.

  │                              │ zurück (sortiert nach Name)               │
  ├──────────────────────────────┼───────────────────────────────────────────┤
  │ create({...})                │ Legt neuen User an, Passwort wird mit     │
  │                              │ bcrypt (Kostenfaktor 12) gehasht          │
  ├──────────────────────────────┼───────────────────────────────────────────┤
  │ update(id, {...})            │ Aktualisiert Userdaten (kein Passwort)    │
  ├──────────────────────────────┼───────────────────────────────────────────┤

  │                              │ (Datensatz bleibt in DB erhalten)         │

  │ updatePassword(id, newPw)    │ Setzt neues Passwort (gehasht mit bcrypt) │

  ├──────────────────────────────┼───────────────────────────────────────────┤

  └──────────────────────────────┴───────────────────────────────────────────┘

  ── models/box.js ─────────────────────────────────────────────────────────

  Tabelle: boxes
  Speichert alle technischen Konfigurationsdaten einer Glovebox.
  Wichtige Felder: manufacturer, project_number (eindeutiger Schlüssel),
  box_type, box_alias, has_o2_sensor, has_h2o_sensor, has_fridge,
  has_oil_pump, has_solvent_filter, last_h2o_cleaning, last_charcoal_done,
  last_sieve_done, last_solvent_test, last_oil_done (Wartungsdaten), is_active

  ┌─────────────────────────────────────┬──────────────────────────────────────┐
  │ findAllByCompany(companyId)         │ Alle aktiven Boxen eines Unternehmens │
  ├─────────────────────────────────────┼──────────────────────────────────────┤
  │ findById(id, companyId)             │ Eine Box (mit Firmenzugehörigkeits-   │
  │                                     │ prüfung, damit kein fremder Zugriff)  │
  ├─────────────────────────────────────┼──────────────────────────────────────┤
  │ create(data)                        │ Legt neue Box an mit allen Konfig-    │
  │                                     │ feldern. Setzt alle Wartungsdaten     │
  │                                     │ initial auf GETDATE() (heute)         │
  ├─────────────────────────────────────┼──────────────────────────────────────┤
  │ update(id, companyId, data)         │ Aktualisiert Basisdaten der Box       │
  │                                     │ (Hersteller, Typ, Alias, Notizen)     │
  ├─────────────────────────────────────┼──────────────────────────────────────┤
  │ softDelete(id, companyId)           │ Setzt is_active=0 (kein echtes        │
  │                                     │ Löschen, Daten bleiben erhalten)      │
  ├─────────────────────────────────────┼──────────────────────────────────────┤
  │ updateMaintenanceDate(id, field)    │ Setzt ein Wartungsdatum-Feld auf      │

  │                                     │ klickt (z.B. last_h2o_cleaning)       │
  └─────────────────────────────────────┴──────────────────────────────────────┘

  ── models/company.js ─────────────────────────────────────────────────────

  Tabelle: companies
  Speichert Unternehmensadresse und Typ (university / startup / company).

  ┌────────────────────┬──────────────────────────────────────────────────────┐
  │ create({...})      │ Legt neues Unternehmen an, gibt die neue ID zurück   │
  ├────────────────────┼──────────────────────────────────────────────────────┤
  │ findById(id)       │ Gibt ein Unternehmen anhand seiner ID zurück         │
  └────────────────────┴──────────────────────────────────────────────────────┘

  ── models/measurement.js ─────────────────────────────────────────────────

  Tabelle: measurements
  Speichert jeden Messwert-Eintrag mit Zeitstempel und User-Kürzel.

  ├────────────────────────────┼────────────────────────────────────────────────┤
  │ findByBox(boxId, months)   │ Gibt alle Messwerte einer Box der letzten X    │

════════════════════════════════════════════════════════════════════════════════
  HINWEISE FÜR WEITERENTWICKLUNG
════════════════════════════════════════════════════════════════════════════════

  - Reset-Tokens (Passwort vergessen) sind aktuell im RAM (Map).
    Bei Server-Neustart gehen offene Links verloren → für Produktion in DB.

  - Die Ampellogik (grün/gelb/rot) liegt als pure Funktion in
    services/alerts.js (buildAlerts/overallStatus) und wird in
    routes/monitoring.js aufgerufen; Ergebnis geht als "alerts" +
    "statusColor" an die View. Regeln: ppm >= 5 gelb, >= 10 rot (quittierbar
    über alert_acks-Tabelle), 2000h-H2O-Reinigung (= Stunden seit
    last_h2o_cleaning, Box läuft 24/7), Aktivkohle-/Molsieb-Zyklen laut
    Boxconfig, 6-Monats-Regeln für Lösungsmittelsensor-Test und Ölwechsel.
    "Erledigt"-Klick setzt das jeweilige Wartungsdatum auf heute
    (POST /monitoring/:id/resolve/:field) bzw. quittiert ppm-Meldungen
    (POST /monitoring/:id/ack/:key). Rote Meldungen stehen immer zuerst.

  - Neue Routen immer in server.js einbinden und mit requireAuth schützen.

  - Neue datenbankgebundene Funktionen gehören ins zugehörige Model,
    nicht direkt in die Route.

  - Bei Cloud-Deployment (Render, Heroku, Railway etc.) immer
    app.set('trust proxy', 1) in server.js behalten – sonst funktionieren
    Session-Cookies im Produktionsmodus nicht (SSL-Terminierung am Proxy).

  - Azure SQL pausiert sich im Serverless-Tier automatisch. Für Produktion
    auf "Provisioned" wechseln oder Firewall-Regel für alle IPs setzen
    (0.0.0.0 – 255.255.255.255) damit Verbindungen nicht geblockt werden.

  - Diese Datei bei jeder neuen Funktion, Route oder View aktualisieren.
